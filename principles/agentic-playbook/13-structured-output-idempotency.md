# Structured Output + Idempotency – Agentic Instruction Sheet

## What It Is
The mandate to force LLMs to communicate exclusively via strictly typed, schema-validated formats (like JSON) and to design tools so that repeated identical executions do not cause unintended side effects.

## Why It Matters for Agentic Systems
Parsing natural language is fragile. If an agent outputs `{"result": "success"}` one time and `Here is the result: success` the next, downstream code will crash. Furthermore, agents often retry tool calls when confused; if an API is not idempotent, an agent might accidentally charge a customer's credit card three times during a retry loop.

## Core Rules
- Never rely on regex or string matching to parse LLM decisions; use native Structured Outputs (JSON schema enforcement).
- Define all tool inputs and agent responses using strictly typed schemas (e.g., Pydantic).
- Implement idempotency keys in all state-mutating tools to prevent duplicate executions during ReAct retry loops.
- Validate LLM outputs against the schema deterministically before passing data to the next node.
- If validation fails, use an automated feedback loop to pass the schema error back to the LLM for correction.

## How to Apply It
1. Define your desired output using a Pydantic model.
2. Use the provider's native structured output feature (e.g., OpenAI `response_format={"type": "json_schema"}`) to guarantee adherence.
3. For all tools that perform actions, require a unique `idempotency_key` generated by the agent or graph.
4. In the tool backend, check if the `idempotency_key` has already been processed. If yes, return the cached success message instead of re-executing.

## Agentic Patterns & Ready-to-Use Examples

**Enforcing Structured Output (Pydantic & OpenAI)**
```python
from pydantic import BaseModel
import openai

class FinalDecision(BaseModel):
    is_approved: bool
    reason: str
    confidence_score: float

# Native schema enforcement (Zero parsing errors)
response = client.beta.chat.completions.parse(
    model="gpt-4o",
    messages=[{"role": "user", "content": "Review this claim..."}],
    response_format=FinalDecision
)
decision = response.choices[0].message.parsed
print(decision.is_approved) # Deterministic boolean
```

**Idempotent Tool Execution**
```python
def charge_customer(amount: float, idempotency_key: str):
    # Backend check prevents double-charging if agent retries
    if db.has_processed(idempotency_key):
        return "Transaction already processed successfully."

    payment_gateway.charge(amount)
    db.save_processed(idempotency_key)
    return "Transaction successful."
```

## Common Pitfalls & How to Avoid Them

- **String Parsing Failures:** Using .split("Final Answer:") to extract data. Fix: Use API-native Structured Outputs.
- **Accidental Duplication:** The LLM receives a timeout error, retries the send_email tool, and spams the user. Fix: Mandate idempotency keys for all action tools.
- **Schema Hallucination:** The LLM makes up fields that aren't in the schema. Fix: Turn on strict: true in OpenAI JSON schemas or use structured generation libraries like Instructor/Outlines.

## Quick Checklist (for self-review or automated eval)

- Do all agents return strictly typed JSON/Pydantic objects instead of raw text?
- Are all state-mutating tools protected by idempotency checks?
- Is schema validation failure handled gracefully (e.g., auto-retry with error message)?
- Have you eliminated all brittle regex/string manipulation for extracting agent intent?
